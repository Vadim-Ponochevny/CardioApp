# 1. Общая информация

## 1.1 Название проекта

**Мобильное приложение «Дневник сопровождения»** (рабочее имя в коде: **CardioApp**).

## 1.2 Цель проекта

Создать мобильную платформу для семей детей после кардиохирургических операций, которое заменяет бумажный дневник наблюдений и позволяет:

- ежедневно фиксировать показатели состояния (давление, дыхание, пульс, кислород и МНО по необходимости);
- отмечать приём назначенных препаратов(витаминок);
- проходить регулярные опросы через Яндекс Формы;
- быстро находить контакты помощи и ответы на частые вопросы;
- получать понятные предупреждения при выходе показателей за медицински согласованные границы.

Главная задача — упростить сбор данных, снизить потери информации и помочь семье вовремя обратиться за помощью.

## 1.3 Термины и определения

| Термин | Значение |
| --- | --- |
| Пользователь | Человек, использующий приложение на устройстве (родитель или ребёнок) |
| Пациент | Ребёнок, за которого ведётся дневник наблюдений |
| Дневник | Совокупность записей показателей, отметок приёма витаминок и ответов на опросы |
| Запись показателей | Одна фиксация набора измерений за конкретные дату и время |
| Показатель | Измеряемый параметр: давление, дыхание, пульс, кислород (SpO2) |
| Витаминка | Назначенный пользователем препарат; в интерфейсе не используется слово «препарат» |
| Возрастная группа | Диапазон возраста пациента, определяющий границы показателей |
| Границы показателей | Норма, зона «обратить внимание», зона «обратиться к врачу в ближайшее время» |
| Локальный профиль | Настройки текущего пользователя приложения (DataStore), не справочник всех пациентов центра |
| MVP | Минимально жизнеспособная версия для пилота |

## 1.4 Границы проекта

### Входит в MVP

- Android-приложение для смартфонов;
- экран «Сегодня» с карточками показателей, витаминок и опроса;
- пошаговое создание и редактирование записей показателей;
- CRUD витаминок и отметка приёма за день;
- раздел «История» с просмотром истории записей по «Показателям» и приемов «Витаминок»;
- раздел «Помощь» с телефонами и FAQ;
- WebView для прохождения опроса (Яндекс Формы);
- локальное хранение данных (Room + DataStore);
- проверка показателей по границам для тестовой возрастной группы 10–11 лет.

### Не входит в MVP

- серверная часть и REST API;
- авторизация и регистрация пользователей;
- синхронизация с Firebase и облачным хранилищем;
- push-уведомления и напоминания;
- интеграция с ЕГИСЗ и медицинскими информационными системами;
- телемедицина и видеозвонки;
- веб-кабинет для врача;
- iOS-версия;
- полный набор возрастных групп (на старте — одна тестовая группа);
- учёт внезапных симптомов и жалоб (отложено, см. открытые вопросы);
- автоматическая загрузка ссылки на опрос из backend.

# 4. Пользовательские сценарии (Use Cases)

## UC-01 Открытие приложения и переход на «Сегодня»

**Участники:** родитель или ребёнок.

**Предусловия:** приложение установлено; локальный профиль пациента задан (для MVP — тестовый профиль).

**Основной сценарий:**

1. Пользователь открывает приложение.
2. Отображается главный экран «Сегодня» с тремя карточками.
3. В нижней навигации доступны разделы «Сегодня», «История», «Помощь».

**Результат:** пользователь видит актуальное состояние дня.

---

## UC-02 Создание записи показателей

**Участники:** родитель или ребёнок.

**Предусловия:** пользователь на экране «Сегодня».

**Основной сценарий:**

1. Пользователь нажимает «Добавить запись» в карточке показателей.
2. Последовательно заполняет 4 шага: давление → дыхание → пульс → кислород.
3. На каждом шаге видит заголовок показателя, поле ввода, прогресс «N из 4», кнопку «Далее».
4. При вводе система проверяет значение по границам возрастной группы и при необходимости показывает предупреждение.
5. После 4-го шага открывается экран «Молодец!» с кнопкой «Завершить».
6. Пользователь возвращается на «Сегодня»; счётчик записей за день увеличивается.

**Исключения:**

- пользователь нажал «Назад» — возврат на предыдущий шаг или выход из flow с подтверждением;
- введено некорректное значение — поле не принимает ввод или показывает ошибку формата.

**Результат:** запись сохранена в локальной БД.

---

## UC-03 Просмотр и редактирование записей за сегодня

**Участники:** родитель или ребёнок.

**Предусловия:** есть хотя бы одна запись за текущий день.

**Основной сценарий:**

1. Пользователь нажимает «Записи за сегодня».
2. Видит список записей за текущий день.
3. Нажимает на запись — открываются детали со всеми показателями.
4. Нажимает «Изменить» у нужного показателя — открывается экран ввода одного показателя.
5. Сохраняет изменения кнопкой «Сохранить».

**Результат:** запись обновлена; предупреждения по границам пересчитаны.

---

## UC-04 Отметка приёма витаминок

**Участники:** родитель или ребёнок.

**Предусловия:** в списке есть хотя бы одна витаминка.

**Основной сценарий:**

1. На карточке «Витаминки» пользователь видит список и прогресс «N из M».
2. Ставит чекбокс напротив принятой витаминки.
3. Прогресс обновляется.

**Результат:** факт приёма сохранён для текущего дня.

---

## UC-05 Управление списком витаминок

**Участники:** родитель.

**Основной сценарий:**

1. Пользователь нажимает «Посмотреть витаминки».
2. Видит полный список.
3. Может открыть витаминку, отредактировать название/дозировку, удалить.
4. Через FAB создаёт новую витаминку.

**Результат:** список назначений актуален; для новых витаминок создаются отметки приёма на текущий день.

---

## UC-06 Прохождение опроса

**Участники:** родитель или ребёнок.

**Основной сценарий:**

1. Пользователь нажимает «Пройти опрос» на карточке опроса.
2. Открывается WebView с активной Яндекс Формой.
3. Пользователь заполняет форму и закрывает WebView.

**Исключения:**

- нет интернета — показывается сообщение об ошибке;
- ссылка недоступна — сообщение об ошибке.

**Результат:** опрос пройден на стороне Яндекс Форм (ответы не хранятся локально в MVP).

---

## UC-07 Просмотр истории

**Участники:** родитель.

**Основной сценарий:**

1. Пользователь переходит в раздел «История».
2. Переключает «Показатели» или «Витаминки».
3. Просматривает записи за прошлые дни (детализация списка — базовая в MVP).

**Результат:** пользователь видит накопленную историю из локальной БД.

---

## UC-08 Получение помощи

**Участники:** родитель или ребёнок.

**Основной сценарий:**

1. Пользователь переходит в раздел «Помощь».
2. Видит список телефонов с пояснениями.
3. Может скопировать номер телефона
4. Просматривает блок вопросов и ответов.

**Результат:** пользователь нашёл нужный контакт или ответ.

| Протокол | HTTPS |
| Формат | HTML-форма в WebView |
| Хранение ответов | На стороне Яндекс Форм; в MVP локально не сохраняются |
| Ошибки | Нет сети, недоступная ссылка — сообщение пользователю |

## 7.2 Будущие интеграции (вне MVP)

| Система | Назначение | Статус |
| --- | --- | --- |
| Firebase / backend | Синхронизация профиля и записей | Запланировано, не проектируется в MVP |
| Push-сервис | Напоминания о замерах и витаминках | Следующий этап |
| Remote config | Обновление ссылки на опрос | Следующий этап |

# 8. Архитектура решения

## 8.1 Общая схема (MVP)

```text
┌─────────────────────────────────────────┐
│              :app                        │
│  MainActivity, NavHost, Theme, Hilt     │
└───────────────┬─────────────────────────┘
                │
    ┌───────────┼───────────┬──────────────┐
    ▼           ▼           ▼              ▼
:feature:   :feature:   :feature:     :feature:
  today    healthrecords vitamins    history/help
    │           │           │              │
    └───────────┴───────────┴──────────────┘
                │
    ┌───────────┼───────────┐
    ▼           ▼           ▼
:core:ui   :core:domain  :core:model
                │
                ▼
           :core:data
         /            \
   Room (DAO)    DataStore
   :core:database
```

## 8.2 Архитектурные принципы

- **Clean Architecture** + **MVVM** + **Jetpack Compose**;
- UI описывается immutable `UiState`; события идут во `ViewModel`;
- `domain` не зависит от Android;
- репозитории — интерфейсы в `domain`, реализации в `data`;
- use case только при наличии нетривиальной бизнес-логики (например, проверка границ показателей);
- DI через **Dagger Hilt**.

# 10. UX/UI требования

- Детский дружелюбный интерфейс: простые формулировки, «витаминки» вместо «препараты».
- Интерфейс должен быть удобен родителю, который часто действует от имени ребёнка.
- Главный экран — «Сегодня»; нижняя навигация: «Сегодня», «История», «Помощь».
- Карточки на главном экране визуально разделены по функциям.
- Пошаговый ввод показателей с явным прогрессом «N из 4».
- Экран успеха после создания записи — позитивная обратная связь («Молодец!»).
- Предупреждения о выходе за границы — спокойные, без паники; формулировки согласуются с медицинской стороной.
- Единая дизайн-система в `:core:ui` (цвета, типографика, кнопки, карточки, поля ввода).
- Пустые состояния (нет записей, нет витаминок) — с понятным призывом к действию.

# 11. Ограничения

## 11.1 Технические

- Только Android; iOS не разрабатывается в рамках данного ТЗ.
- MVP работает офлайн для дневниковых данных; для опроса нужен интернет.
- Один локальный профиль пациента на установку приложения.
- Одна тестовая возрастная группа (10–11 лет) с полными границами в MVP.

## 11.2 Организационные

- Тексты предупреждений и FAQ требуют согласования с представителем лаборатории.
- Ссылка на Яндекс Форму задаётся вручную (seed / конфиг) до появления backend.
- Пилот: 5–10 тестовых семей (по паспорту проекта).

## 11.3 Медицинские и юридические

- Приложение — вспомогательный инструмент наблюдения, не медицинское изделие в рамках MVP.
- Не использовать формулировки, создающие у пользователя впечатление постановки диагноза.

# 12. Критерии приёмки

## 12.1 Экран «Сегодня»

Считается выполненным, если:

- отображаются три карточки (показатели, витаминки, опрос);
- счётчик записей за день корректен;
- чекбоксы витаминок работают и обновляют прогресс;
- нижняя навигация переключает разделы.

## 12.2 Записи показателей

Считается выполненным, если:

- создаётся запись через 4 шага;
- запись сохраняется и отображается в «Записи за сегодня»;
- редактирование отдельного показателя работает;
- при значении вне нормы показывается предупреждение (для пульса, давления, SpO2);
- экран «Молодец!» возвращает на «Сегодня».

## 12.3 Витаминки

Считается выполненным, если:

- CRUD витаминок работает;
- удаление убирает витаминку из списка;
- отметка приёма сохраняется и переживает перезапуск приложения.

## 12.4 Опрос

Считается выполненным, если:

- WebView открывает активную ссылку;
- при отсутствии сети показывается ошибка.

## 12.5 История и помощь

Считается выполненным, если:

- переключатель «Показатели / Витаминки» работает;
- контакты отображаются и открывают dialer;
- FAQ отображается.

## 12.6 Данные и архитектура

Считается выполненным, если:

- профиль хранится в DataStore;
- дневниковые данные — в Room;
- границы 10–11 лет предзаполнены;
- основные экраны имеют Compose Preview.
