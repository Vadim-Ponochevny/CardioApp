# CardioApp: проектный ориентир

Документ фиксирует текущее понимание MVP, архитектуры и структуры Android-приложения для сопровождения детей после кардиохирургических операций. Его нужно обновлять после архитектурных решений, изменения модулей, схемы БД и пользовательских сценариев.

## Контекст

Приложение помогает семье вести дневник наблюдения после выписки: записывать показатели, отмечать прием "витаминок", проходить регулярные опросы и быстро находить контакты помощи. Интерфейс проектируется как детский и дружелюбный, но должен оставаться удобным для родителя, который часто действует от имени ребенка.

Источник продуктового контекста: `.cursor/docs/НИИ_КПСЗ_Паспорт_проекта_МП_Дневник_сопровождения.md`.

## Терминология в коде

Слово `Vital` в первом наброске было взято от английского `vital signs` - жизненные показатели. Для проекта оно может быть неочевидным, поэтому дальше используем более прямые названия:

- `HealthRecord` - одна запись показателей за конкретные дату и время;
- `HealthMetric` или `MetricType` - тип показателя, например давление, дыхание, пульс, кислород;
- `HealthMetricLimits` - границы показателей для возрастной группы.

В пользовательском интерфейсе говорим просто `показатели` и `записи`.

## MVP

### Нижняя навигация

В MVP есть три основных раздела:

- `Сегодня` - главный ежедневный экран.
- `История` - история показателей и приемов витаминок.
- `Помощь` - контакты и вопросы-ответы.

### Сегодня

Главный экран называется в интерфейсе `Сегодня` и содержит три карточки:

- Карточка показателей:
  - количество записей за сегодня;
  - кнопка `Записи за сегодня` открывает список дневных записей;
  - кнопка `Добавить запись` запускает пошаговое создание записи.
- Карточка витаминок:
  - список назначенных витаминок;
  - чекбокс слева у каждой витаминки для отметки приема;
  - прогресс вида `1 из 2`;
  - кнопка `Посмотреть витаминки` открывает полный список витаминок.
- Карточка опроса:
  - кнопка `Пройти опрос`;
  - открывает WebView с Яндекс Формой.

### Создание записи показателей

Создание записи - пошаговый flow из четырех экранов:

1. Давление.
2. Дыхание.
3. Пульс.
4. Кислород.

Общий UI шага:

- сверху кнопка назад слева;
- по центру заголовок текущего показателя;
- поле ввода значения;
- над нижней кнопкой прогресс `1 из 4`;
- нижняя кнопка `Далее`.

После четвертого шага открывается экран успеха с текстом в духе `Молодец!` и кнопкой `Завершить`, которая возвращает пользователя на `Сегодня`.

### Записи за сегодня

Экран показывает все записи показателей за текущий день. На запись можно нажать, после чего открывается экран деталей записи:

- показатели отображаются списком;
- под каждым показателем есть кнопка `Изменить`;
- редактирование открывает экран ввода конкретного показателя;
- в режиме редактирования нет прогресса `1 из 4`;
- основная кнопка называется `Сохранить`.

### Витаминки

Из карточки на главном экране открывается список всех витаминок:

- можно открыть витаминку и посмотреть детали;
- можно отредактировать данные витаминки;
- можно создать новую витаминку через FAB;
- удалить витаминку можно на экране деталей витаминки.

В пользовательском интерфейсе и в коде используем слово `витаминки` / `Vitamin`, не `препараты` / `Medication`.

### История

В верхней части экрана истории есть переключатель:

- `Показатели`;
- `Витаминки`.

Детализацию истории делаем позже. На MVP достаточно заложить отдельный раздел и доменные модели так, чтобы историю можно было строить по локальной БД.

### Помощь

Раздел содержит:

- номера телефонов для связи;
- блок частых вопросов и ответов.

## Пользовательские данные

Минимальный профиль пользователя:

- `patientId` - идентификатор пациента;
- `birthDate` - дата рождения;
- `ageGroup` - возрастная группа;
- `userType` - тип пользователя: родитель или ребенок.

Возрастная группа нужна для проверки границ показателей. Для MVP можно использовать одного тестового пользователя и один набор ограничений, но модель должна позволять заменить ограничения на групповые.

## Локальное хранение

Основной источник дневниковых данных MVP - локальная БД Room. Профиль текущего пользователя не хранится в Room: это локальная настройка/профиль приложения, который хранится через DataStore.

Предварительные сущности:

- `HealthRecordEntity` - одна запись показателей за дату и время.
- `SingleMetricLimitEntity` - границы одиночных показателей для возрастной группы.
- `BloodPressureLimitEntity` - границы давления для возрастной группы.
- `VitaminEntity` - назначенная витаминка.
- `VitaminIntakeDayEntity` - дневной лист приема витаминок.
- `VitaminIntakeEntity` - факт приема конкретной витаминки в рамках дневного листа.
- `SurveyLinkEntity` - актуальная ссылка на Яндекс Форму.
- `HelpContactEntity` - контакт помощи.
- `FaqEntity` - вопрос и ответ.

### Схема БД MVP

Версия схемы на старт: `1`.

Картинка схемы: `.cursor/docs/db-schema.svg`.

#### Локальный профиль пользователя

Профиль текущего пользователя хранится не в Room, а в DataStore. Это не справочник пациентов центра, а локальная настройка текущего установленного приложения.

Минимальные поля профиля:

| Поле | Тип | Описание |
| --- | --- | --- |
| `patientExternalId` | `String` | Id, выданный пациенту/семье центром. |
| `birthDate` | `String` | Дата рождения в формате `yyyy-MM-dd`. |
| `ageGroup` | `String` | Возрастная группа, например `Age10To11`. |
| `userType` | `String` | `Parent` или `Child`. |

Позже рядом может появиться технический `firebaseUserId`, но Firebase пока не проектируем и не добавляем.

#### `health_records`

Одна запись показателей за дату и время.

| Поле | Тип | Описание |
| --- | --- | --- |
| `id` | `String` | Primary key, UUID. |
| `patientId` | `String` | Внешний id пациента из локального профиля. Не foreign key. |
| `createdAt` | `Instant` as `Long` | Когда запись создана / к какому времени относится измерение. |
| `updatedAt` | `Instant` as `Long` | Когда запись изменялась последний раз. |
| `systolicPressure` | `Int` | Верхнее давление. |
| `diastolicPressure` | `Int` | Нижнее давление. |
| `respiratoryRate` | `Int` | Дыхание, вдохов в минуту. |
| `heartRate` | `Int` | Пульс, ударов в минуту. |
| `oxygenSaturation` | `Int` | SpO2, проценты. |

Индексы:

- `patientId`;
- `createdAt`;
- составной индекс `patientId, createdAt` для экранов `Сегодня` и `История`.

#### `single_metric_limits`

Справочник границ одиночных показателей по возрастной группе. Для MVP засеяны значения для `Age10To11`.

| Поле | Тип | Описание |
| --- | --- | --- |
| `id` | `String` | Primary key, например `age-10-11-heart-rate`. |
| `ageGroup` | `String` | Возрастная группа. |
| `metricType` | `String` | `HeartRate`, `OxygenSaturation`, `RespiratoryRate`. |
| `normalMin` | `Int` | Нижняя граница нормы. |
| `normalMax` | `Int` | Верхняя граница нормы. |
| `attentionMin` | `Int?` | Нижняя граница зоны внимания. |
| `attentionMax` | `Int?` | Верхняя граница зоны внимания. |
| `doctorSoonMin` | `Int?` | Нижняя граница критической зоны, если применимо. |
| `doctorSoonMax` | `Int?` | Верхняя граница критической зоны, если применимо. |

Индексы:

- уникальный индекс `ageGroup, metricType`.

#### `blood_pressure_limits`

Отдельный справочник границ давления, потому что давление состоит из двух значений: верхнего и нижнего.

| Поле | Тип | Описание |
| --- | --- | --- |
| `id` | `String` | Primary key, например `age-10-11-blood-pressure`. |
| `ageGroup` | `String` | Возрастная группа. |
| `normalSystolicMin` | `Int` | Нижняя граница верхнего давления. |
| `normalSystolicMax` | `Int` | Верхняя граница верхнего давления. |
| `normalDiastolicMin` | `Int` | Нижняя граница нижнего давления. |
| `normalDiastolicMax` | `Int` | Верхняя граница нижнего давления. |
| `doctorSoonSystolicLow` | `Int` | Верхнее давление ниже этого значения - повод обратиться к врачу. |
| `doctorSoonSystolicHigh` | `Int` | Верхнее давление выше этого значения - повод обратиться к врачу. |
| `doctorSoonDiastolicLow` | `Int` | Нижнее давление ниже этого значения - повод обратиться к врачу. |
| `doctorSoonDiastolicHigh` | `Int` | Нижнее давление выше этого значения - повод обратиться к врачу. |

Индексы:

### Запись показателей

Для MVP не делаем отдельную таблицу под каждый конкретный показатель. Набор показателей фиксированный, поэтому запись удобнее хранить одной сущностью:

- `id`;
- `patientId`;
- `createdAt`;
- `systolicPressure` - верхнее давление;
- `diastolicPressure` - нижнее давление;
- `respiratoryRate` - дыхание, вдохов в минуту;
- `heartRate` - пульс, ударов в минуту;
- `oxygenSaturation` - кислород, проценты SpO2.

Все записи хранятся в базе. Экран `Записи за сегодня` получает только записи текущего дня через фильтр по `createdAt`.

Отдельная сущность для значений может понадобиться позже, если набор показателей станет динамическим. Сейчас это лишняя гибкость и усложнение схемы.

### Витаминки

`VitaminEntity` хранит список назначенных витаминок, который пользователь задает сам. Если витаминка есть в списке, считаем ее активной. Отдельные поля `comment`, `isActive`, `createdAt`, `updatedAt` для MVP не нужны.

`VitaminIntakeDayEntity` отражает дневной лист приема:

- `id`;
- `patientId`;
- `date`.

`VitaminIntakeEntity` отражает отметку приема конкретной витаминки за день:

- `id`;
- `intakeDayId`;
- `vitaminId`;
- `isTaken`;
- `takenAt`.

При открытии карточки или экрана витаминок приложение берет актуальный список `VitaminEntity` пациента и состояние приема за выбранный день. Если для дня еще нет записей приема, они создаются или собираются в UI как непомеченные.

### Опросы

`SurveyLinkEntity` нужен, потому что ссылка на Яндекс Форму может меняться:

- `id`;
- `url`;
- `title`;
- `isActive`;
- `updatedAt`.

Для MVP можно иметь одну активную ссылку. Позже можно добавить загрузку ссылки из удаленной конфигурации или backend.

### Границы показателей

Границы зависят от возрастной группы пациента. Для MVP используем тестовую группу `10-11 лет`, но модель и use case должны быть написаны так, чтобы позже добавить все группы.

Источник: `.cursor/docs/Критические значения КХО2.md`.

На старте можно не делать полноценный редактор или загрузку границ из сети. Достаточно засеять локальную БД тестовыми значениями для `10-11 лет`: одиночные показатели попадают в `single_metric_limits`, давление - в `blood_pressure_limits`.

Тестовые значения для `10-11 лет`:

- Пульс:
  - норма: `60-80`;
  - обратить внимание: `45-60`;
  - обратиться к врачу в ближайшее время: `45 и ниже`.
- Давление:
  - норма: `110-125 / 70-85`;
  - обратиться к врачу в ближайшее время: ниже `110/70` или выше `125/85`.
- Кислород SpO2:
  - норма: `95-100%`;
  - обратить внимание: `90-95%`;
  - обратиться к врачу в ближайшее время: ниже `90%`.
- Дыхание:
  - в текущем документе с критическими значениями не найдено;
  - до уточнения храним значение, но не показываем медицинскую оценку по границам.

В коде это представлено как sealed-модель правил по возрастной группе:

- `SingleMetricLimits` - пульс, кислород, дыхание;
- `BloodPressureLimits` - верхнее и нижнее давление.

Для давления диапазон состоит из двух частей: верхнее и нижнее давление. Для остальных показателей достаточно одного числового значения.

Проверка показателей:

- при вводе значения приложение сравнивает показатель с границами для возрастной группы;
- если значение выходит за пределы, UI показывает понятное предупреждение;
- приложение не ставит диагноз и не заменяет врача;
- формулировки предупреждений нужно отдельно согласовать с медицинской стороной.

## Архитектура

Цель: Clean Architecture + MVVM + Jetpack Compose.

Основные правила:

- UI строится на Compose.
- Каждый экран управляется `ViewModel`.
- `ViewModel` работает с repository-интерфейсами и use case только там, где есть реальная бизнес-логика.
- `domain` не содержит Android-зависимостей.
- `data` содержит Room, DataStore, DAO, entity, mapper и реализации репозиториев.
- Навигация живет на уровне `app` и feature API.
- Состояние UI описывается immutable `UiState`.
- События пользователя идут во `ViewModel` через методы или action/event модели.
- Не делать use case-обёртки, которые просто вызывают один метод repository.

## Модульная структура

Ориентир - подход Now in Android: `app`, `core`, `feature`. Но для MVP не дробим проект слишком мелко.

Целевая структура на старт:

```text
:app
:core:common
:core:model
:core:database
:core:data
:core:domain
:core:ui
:feature:today
:feature:healthrecords
:feature:vitamins
:feature:history
:feature:help
```

### Ответственность модулей

`:app`

- `MainActivity`;
- приложение, тема верхнего уровня, nav host;
- сборка feature-модулей в единый граф навигации;
- DI entry point.

`:core:common`

- общие Kotlin-утилиты;
- работа с датой и временем;
- базовые result/error модели, если понадобятся.

`:core:model`

- доменные модели без Android-зависимостей;
- `Patient`, `AgeGroup`, `UserType`;
- `HealthRecord`, `MetricType`, `HealthMetricLimits`;
- `Vitamin`, `VitaminIntake`;
- модели помощи и FAQ.

`:core:database`

- Room database;
- entity;
- DAO;
- миграции.

`:core:data`

- реализации репозиториев;
- mapper между entity и domain model;
- локальные data source.

`:core:domain`

- repository-интерфейсы;
- use case только там, где есть реальная бизнес-логика;
- бизнес-правила проверки показателей.

`:core:ui`

- дизайн-система;
- тема, цвета, типографика;
- общие Compose-компоненты: карточки, кнопки, поля ввода, empty/error states.

`:feature:today`

- экран `Сегодня`;
- карточки показателей, витаминок и опроса;
- переходы в записи, витаминки и WebView опроса.

`:feature:healthrecords`

- flow создания записи показателей;
- список записей за сегодня;
- детали записи;

## Навигация

Основные top-level destination:

- `TodayRoute`;
- `HistoryRoute`;
- `HelpRoute`.

Внутренние destination:

- `TodayRecordsRoute` - записи за сегодня;
- `HealthRecordCreateRoute` - создание записи;
- `HealthRecordSuccessRoute` - экран успеха;
- `HealthRecordDetailsRoute(recordId)` - детали записи;
- `HealthMetricEditRoute(recordId, metricType)` - редактирование одного показателя;
- `VitaminsRoute` - список витаминок;
- `VitaminDetailsRoute(vitaminId)` - детали витаминки;
- `VitaminEditRoute(vitaminId?)` - создание или редактирование;
- `SurveyWebViewRoute` - WebView с Яндекс Формой.

Для MVP можно начать с обычного Navigation Compose. Если будет принято решение использовать Navigation 3, нужно отдельно зафиксировать back stack, entry provider и top-level navigation.

## Build Variants и Flavors

Нужно заложить три flavor:

- `dev` - основная разработка, локальная БД, можно включать debug-инструменты и тестовые данные.
- `demo` - демонстрационный режим с предзаполненными моковыми данными для презентации экранов без ручного ввода.
- `prod` - production-конфигурация без моков и debug-поведения.

Предварительные варианты сборки:

- `devDebug` - ежедневная разработка;
- `demoDebug` - быстрая демонстрация MVP;
- `prodRelease` - будущая релизная сборка.

Для экранов Compose нужно делать `@Preview`. Превью не должны ходить в Room или реальные репозитории. Для этого:

- UI-компоненты принимают `UiState` и callbacks;
- в feature-модулях создаются preview/mock состояния;
- для сложных экранов можно держать `PreviewParameterProvider`;
- demo flavor может использовать fake repository или seed-данные.

## Первый порядок реализации

1. Настроить базовую модульную структуру Gradle.
2. Завести `core:model`, `core:domain`, `core:database`, `core:data`.
3. Описать модели пациента, показателей, витаминок и истории приемов.
4. Подключить Room и создать DAO для MVP.
5. Собрать `:core:ui` с базовой темой и общими компонентами.
6. Реализовать `:feature:today` с моковыми или локальными данными.
7. Реализовать flow создания записи в `:feature:healthrecords`.
8. Реализовать список и CRUD витаминок в `:feature:vitamins`.
9. Добавить базовые экраны `История` и `Помощь`.

## Текущее состояние репозитория

На текущем этапе проект настроен как базовый многомодульный Android-проект:

- есть модуль `:app`;
- включен Kotlin и Jetpack Compose;
- основной экран пока стандартный шаблон `Hello Android`;
- созданы базовые core-модули: `:core:common`, `:core:model`, `:core:domain`, `:core:database`, `:core:data`, `:core:ui`;
- созданы базовые feature-модули: `:feature:today`, `:feature:healthrecords`, `:feature:vitamins`, `:feature:history`, `:feature:help`;
- подключен Room в `:core:database`;
- профиль пользователя хранится через DataStore, а не в Room;
- созданы entity, DAO, `CardioDatabase` и seed-данные для тестовой возрастной группы `10-11 лет`;
- `:core:data` содержит mappers и repository implementations;
- `:core:domain` содержит repository-интерфейсы и use case только для реальной бизнес-логики проверки границ;
- `:feature:today` содержит первый экран `Сегодня` с `ViewModel`, `UiState` и preview;
- `MainActivity` показывает базовый каркас приложения с нижней навигацией;
- Hilt подключен: `CardioApplication` помечен `@HiltAndroidApp`, `MainActivity` - `@AndroidEntryPoint`, зависимости Room/DAO/repository предоставляются через Hilt-модули в `:core:data`.
